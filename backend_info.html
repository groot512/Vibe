1. 동적 웹의 탄생: 방명록 딜레마

1990년대 초반 웹은 미리 만들어진 HTML 파일을 단순히 전달하는 역할에 불과했습니다. 하지만 1993년, 사용자가 직접 글을 남기는 '방명록'이나 '검색 기능'이 필요해지면서 혁명적인 발상의 전환이 일어납니다. 서버가 파일을 그저 보관하는 것을 넘어, 요청이 들어올 때마다 그 자리에서 HTML 파일을 '제조'하기 시작한 것입니다.

2. 백엔드 언어의 5대 뼈대

Java, Python, Node.js 등 수많은 백엔드 언어가 존재하지만, 본질적인 뼈대는 완벽하게 동일합니다. 문법이라는 '방언'의 차이만 있을 뿐, 서버를 제어하는 근본적인 5가지 개념을 익히면 어떤 언어든 쉽게 확장할 수 있습니다.

3. 코드의 진화: MVC 아키텍처

HTML 제조 코드가 길어지면서 관리가 불가능해졌습니다. 이에 개발자들은 역할을 분리하는 MVC 패턴 (Model-View-Controller)을 도입했습니다. 데이터는 데이터베이스(Model)에, 화면은 뷰(View)에, 계산 로직은 컨트롤러(Controller)에 분리하여 유지보수를 극대화했습니다.

4. 모바일 혁명과 API의 시대

스마트폰의 등장은 서버의 역할을 완전히 바꾸어 놓았습니다. 아이폰 앱, 안드로이드 앱, 웹 브라우저 등 다양한 클라이언트가 등장하면서, 서버는 더 이상 특정 화면(HTML)을 렌더링해서 줄 수 없게 되었습니다. 대신, 순수한 데이터(JSON 등)만을 전달하는 API(Application Programming Interface) 창구 역할로 전환되며 프론트엔드와 백엔드가 완벽하게 분리되었습니다.

단일 서버 구조에서 멀티 클라이언트를 지원하는 API 통신으로의 데이터 트래픽 변화 추이

5. 트래픽 폭주 대응: 서버 스케일링

사용자가 수백만 명으로 늘어나면 서버 한 대로는 버틸 수 없습니다. 문제를 해결하기 위해 서버 성능을 업그레이드하는 스케일 업(Scale Up) 방식과, 저렴한 서버 여러 대를 병렬로 연결하는 스케일 아웃(Scale Out) 방식이 등장했습니다. 현대의 클라우드 인프라에서는 주로 확장이 무한대에 가까운 스케일 아웃을 채택합니다.

스케일 업 (Scale Up)

  • 서버 장비 자체의 부품(CPU, RAM)을 업그레이드
  • 물리적인 성능의 한계가 명확함
  • 데이터베이스(DB) 서버 확장에 주로 사용됨

스케일 아웃 (Scale Out)

  • 적당한 성능의 서버를 여러 대 추가하여 로드밸런싱
  • 이론상 무한한 수평 확장 가능
  • 현대 웹(API 서버)의 표준 확장 방식
