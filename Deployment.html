웹 서버와 배포 (Deployment)

로컬호스트(Localhost)를 넘어 전 세계와 연결되는 서비스 기획의 모든 것

💡 서론: 밀실에서 광장으로

내 컴퓨터 화면에서는 완벽하게 작동하는 서비스라도, localhost:3000 주소로는 다른 사람이 접속할 수 없습니다. 로컬호스트는 '지금 이 컴퓨터 자신(127.0.0.1)'을 뜻하는 전 세계적인 약속이기 때문입니다. 배포란 내 컴퓨터의 코드를 24시간 인터넷에 연결된 공용 컴퓨터로 옮겨 진정한 '서비스'로 탈바꿈시키는 과정입니다.

💻

Localhost (내 컴퓨터)

IP: 127.0.0.1
접근성: 나 혼자만 접속 가능
상태: 밀실에 갇힌 코드

외부 접속 불가
🌐

Deployment (배포된 서버)

IP: 123.45.67.89 (예시)
접근성: 전 세계 누구나 접속 가능
상태: 실제 서비스 가동 중

외부 접속 가능

웹 서버와 WAS의 분업 구조

서버 안에서도 효율적인 처리를 위해 분업이 일어납니다. 정적 파일을 빠르게 처리하는 '자판기' 역할의 웹 서버(Nginx)와, 복잡한 로직을 처리해 동적 화면을 생성하는 '바리스타' 역할의 WAS(Web Application Server)가 결합하여 작동합니다.

📱 Client (브라우저)
🚦 Web Server (Nginx / 자판기)
정적 파일 즉시 반환
☕ WAS (Node, Python / 바리스타)
동적 로직 처리
🗄️ Database (데이터 저장소)

배포 기술의 진화

과거 FTP 방식은 파일을 수동으로 복사하여 덮어쓰기 사고나 환경 불일치(제 컴퓨터에선 되는데요?) 문제를 일으켰습니다. 현재는 Git(버전 관리), Docker(환경 표준화), CI/CD(배포 자동화 파이프라인)를 통해 이 문제를 완벽히 해결했습니다.

▲ FTP 방식과 현대적인 CI/CD 배포 방식의 효율성 비교

CI/CD 파이프라인 흐름

코드를 GitHub에 올리는 순간, 공장의 컨베이어 벨트처럼 자동화된 파이프라인이 가동되어 수 분 내에 서비스가 전 세계로 배포됩니다.

1️⃣
코드 작성 & Push
Git을 통해 GitHub 저장소에 변경 이력 저장
2️⃣
CI (지속적 통합)
GitHub Actions가 자동 테스트 실행 및 오류 검출
3️⃣
Build (도커 이미지 생성)
Docker를 이용해 코드를 동일한 환경의 컨테이너로 포장
4️⃣
CD (지속적 배포)
완성된 이미지를 운영 서버(또는 Vercel/AWS)에 자동 배포

내 서비스에 문패 달기: DNS レ코드

도메인을 구매했다면, 그 도메인이 어떤 서버 IP를 가리킬지 연결해야 합니다. DNS(Domain Name System)는 인터넷의 전화번호부 역할을 하며, 기획자도 필수적으로 알아야 할 5가지 핵심 레코드가 있습니다.

🎯

A 레코드

도메인을 특정 IP 주소(예: 12.34.56.78)로 직접 연결하는 '직행버스' 역할을 합니다.

🪧

CNAME 레코드

도메인을 다른 도메인 주소로 연결합니다. Vercel 등 유동 IP 플랫폼에서 이정표로 쓰입니다.

✉️

MX 레코드

도메인으로 들어오는 이메일을 처리할 메일 서버(예: 구글 워크스페이스)를 지정합니다.

🔐

TXT 레코드

도메인 소유권을 증명하거나 발송 이메일의 스팸 방지 보안 인증(SPF 등)에 쓰입니다.

🏢

NS 레코드

해당 도메인의 DNS 관리를 맡을 주체(네임서버)를 지정합니다. 클라우드플레어(Cloudflare)로 이전할 때 필수적으로 변경합니다.

트래픽 폭주 대처: 로드 밸런싱과 오토스케일링

유명 커뮤니티 바이럴 등으로 접속자가 수백 배 증가할 때 서버 다운을 막는 핵심 아키텍처입니다. 부하를 고르게 분산하는 로드 밸런서와 트래픽 양에 따라 서버 댓수를 유동적으로 조절하는 오토스케일링이 결합하여 안정성을 보장합니다.
